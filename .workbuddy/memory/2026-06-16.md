## board 红点不符合实际位置 Bug 排查与修复（2026-06-16）
- **现象**：相机图像中 board 中心红点及四角红点偏离实际标定板位置
- **根本原因**：原 params.yaml 将 board_height/board_width 以及 delta_*、ArucoID、marker_size 等参数放在 YAML 嵌套键（board_size:、marker_center_distance:、ArUco marker:）下，但 cpp 代码 `n_.getParam("board_width", ...)` 只读顶层参数名，导致这些参数全部读不到，回退到硬编码默认值（board_width=0.7, board_height=0.5），与实际尺寸相反
- **修复**：将所有标定板相关参数"拍平"为顶层 YAML 键，删除 board_size:、marker_center_distance:、ArUco marker: 嵌套结构
- **结论**：ROS n_.getParam() 不支持 YAML 嵌套路径，所有参数必须是顶层键

## estimation_tf 硬编码内参 Bug 修复（2026-06-16）
- **现象**：PnP 3d2d 和 Ceres 3d2d 结果明显错误（平移为负值/过大）
- **根本原因1**：`estimation_tf.cc` solvePNP_cv() 硬编码旧内参（617/324/242），未从 ROS 参数读取
- **根本原因2**：`estimation_tf_node.cc` 第93行 K 矩阵也是硬编码旧值
- **修复**：
  1. solvePNP_cv() 增加cameraMatrix和distCoeffs参数，从外部传入
  2. estimation_tf_node.cc 开头从ROS参数服务器读取fx/fy/cx/cy/k1/k2/k3/p1/p2
  3. 构造cv::Mat和Eigen::Matrix3d传递给各求解函数
  4. 同时修正了console输出标签（3d3d ICP原来误标为"3d2d"）
- **注意**：用户需要将修改后的代码同步到Ubuntu机器，同时确保params.yaml已拍平（无嵌套），然后重新catkin_make + 重新采集数据 + 重新标定

## estimation_tf_node.launch 未加载 params.yaml Bug（2026-06-16）
- **现象**：运行estimation_tf_node后打印的内参仍是旧默认值（617/324/242），而非yaml中的新值
- **根本原因**：estimation_tf_node.launch 没有 `<rosparam file="...params.yaml" />` 这一行，而data_frame_extraction_node.launch有
- **结果**：ROS参数服务器里没有yaml参数，n_.param()全部回退到cpp硬编码默认值
- **修复**：在estimation_tf_node.launch中添加 `<rosparam file="$(find lidar2cam_calibration)/cfg/params.yaml" />`
- **重要**：用户还需确保Ubuntu上的params.yaml也是拍平版+正确内参
